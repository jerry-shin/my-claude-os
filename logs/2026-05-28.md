## 회고 포인트
- 슬래시 커맨드를 만들고 → 실제로 호출해 → 커밋까지 끝내는 루프를 한 번 돌려본 게 의미 있었다. 다음 커맨드도 같은 방식(작성 → 즉시 검증)으로 만들면 좋다.
- 첫 커밋과 두 번째 커밋이 1분 간격으로 나뉘었는데, 처음부터 README 변경도 같이 묶었다면 커밋이 1개로 깔끔했을 것. 다음에는 `git status` 단계에서 추적 누락된 파일을 함께 확인하자.
- **추상화 단계의 자연스러운 발전:** 슬래시 커맨드(단일 작업) → 훅(자동 부수 효과) → 에이전트(전문 역할 + 협업)로 한 세션 안에서 단계가 올라갔다. 각 단계가 그 다음 단계의 토대(스킬 → 사용 통계 → 에이전트의 위임 분석 가능)가 되도록 자연스럽게 쌓였다.
- **에이전트 설계 패턴 발견:** 입출력 파일 표준화(`docs/01~05`), 단계별 사용자 체크포인트, 추적 매트릭스(RTM) 등 모든 에이전트에 공통 적용한 규약이 협업 품질을 결정한다. 다음에 새 에이전트를 추가할 때도 같은 규약을 따라야 호환된다.
